## 3. 关键结论：`exitCode 63` 是什么

`render-process-gone` 的 `reason=launch-failed` 时，**`exitCode` 不是业务退出码，而是"平台特定的启动失败错误码"**（[Electron 官方文档](https://www.electronjs.org/docs/latest/api/structures/render-process-gone-details)）。在 Windows 上，它属于 **Chromium 沙箱 / 子进程启动失败族**（`sandbox_types.h` 的 `ResultCode`：`18=SBOX_ERROR_CREATE_PROCESS`，`62=SBOX_ERROR_UNSANDBOXED_PROCESS`，`63≈SBOX_ERROR_LAST` 哨兵）。

**含义：渲染进程在"启动 / 沙箱化创建子进程"这一步就失败了，根本没跑起来——不是页面 JS 崩溃。** 因此排查方向是**进程启动层面（启动参数 / GPU / 沙箱 / 环境）**，而非页面脚本。

> 置信度：`exitCode 63` 属"启动失败族"为高置信；但**究竟对应哪个具名常量无法行级锁定**（Chromium M106 源码在本环境拉不到，且枚举随版本位移）。

## 4. 回归时间线（核心）：Win10 1607 是怎么被"照顾崩"的

判断函数 `getLegacyWindowsGpuWorkaroundInfo()`（main.js 顶部）随版本演变如下。**Win10 1607 的 `os.release()` = `10.0.14393`，major=10、build=14393。**

| 版本 | commit / 日期 | 1607 命中的判断 | 1607 实际启动配置 | 结果 |
|------|--------------|----------------|------------------|------|
| **1.1.2** | `1e1bf5b` 05-30 | `major<10` 才禁GPU → 1607 走 **else** | **保留 GPU** + 全局开关`*` | ✅ **能用（baseline）** |
| 1.1.3 | `1d5fc8a` 06-03 | 同上（`cc20b06` 只给 Win7/8 加 `--disable-gpu`，1607 吃不到） | 同 1.1.2 | 见 §7 辨析 |
| 1.1.4 后 | `cd40a25` 06-08 | `build<=14393` → **禁 GPU**（`win10-1607-ltsb-gpu-test`） | **禁 GPU**（`disable-gpu`+`disable-gpu-compositing`+`in-process-gpu`）+ 全局开关 | ❌ launch-failed |
| 1.1.5 (HEAD) | `14a4379` 06-08 | `build<=14393` → 保留GPU + compat | 保留GPU + compat开关（`no-sandbox` 等，**无** `disable-gpu-sandbox`） | ❌ launch-failed |

`*` 全局开关：早在 **1.0.x**（`adb615d` 引入 `no-sandbox`、`9ef32de` 引入 `RendererCodeIntegrity`/`disable-renderer-backgrounding`/`disable-backgrounding-occluded-windows`）就对**所有系统**生效。`14a4379` 把它们从"全局"改成"仅旧系统（`shouldUseRendererLaunchCompatibility`）"。

### 4.1 关键洞察

1. **1607 本来不需要任何特殊处理**：1.1.2 时它走正常 Win10 分支（保留 GPU），叠加当时的全局兼容开关，**能正常运行**。
2. **是"特殊照顾"把它搞崩的**：1.1.4 先试"禁 GPU"（崩），1.1.5 改试"保留 GPU + 一套 compat 开关"（仍崩），工作区版又加 `disable-gpu-sandbox` + safe-mode（还是崩）。**每一次"修复"都在 1607 这条没问题的路径上动刀**。
3. **`14a4379` 注释的误导**：compat 分支注释写"*恢复 1.1.2 时代已经验证过的 renderer 启动兼容参数*"——但 **1.1.2 从来没有这套"仅旧系统"的开关**（1.1.2 是全局开关 + 1607 保留 GPU）。这套"compat 开关"是**新造的、从未在 1607 上验证过**的配置。

```
   │         → app.relaunch(--yyzs-renderer-safe-mode) + app.exit(0)
   │
   ├─ 重启进 safe-mode → 命中【禁 GPU】分支 (disable-gpu + in-process-gpu)
   │     → 而"1607 禁 GPU 同样 launch-failed"（注释 main.js:37 + cd40a25 教训）
   │     → 【第二次崩】
   │
   ├─ 第二次 maybeRelaunch 返回 false（isRendererSafeMode 已 true）
   │     → dialog.showErrorBox('启动失败')          →  弹框【"系统错误请重试"】
   │     → app.exit(63)
   │
   └─ 用户再开 → 重复上述全过程                       →  【关闭重启后打不开】
```

**两种模式（保留GPU+compat开关 / 禁GPU）在这台 1607 上都 launch-failed**，safe-mode 只是把两种死法串成了死循环。

## 6. 根因分析（确证 vs 推测，分级）

### ✅ 已确证（代码 + git + 官方文档）
1. `launch-failed` 时 exitCode 是平台启动失败码，渲染进程没起来（非页面崩溃）。
2. 1607 在 1.1.2（保留 GPU + 全局兼容开关）能用；1.1.4(禁GPU)/1.1.5(compat) 把它特殊处理后 launch-failed。
3. safe-mode 重启切到**禁 GPU**，而 1607 禁 GPU 也 launch-failed → 二次崩 + 死循环（解释"重启打不开"）。
4. Electron 21=Chromium 106 **官方仍支持 Win10 1607**（收紧到 Win10-only 是 Electron 23/Chromium 110）。

### 🟠 推测（medium 置信，需真机验证）
- **1.1.5 给 1607 新增的 `disable-gpu-sandbox` 是"压垮的最后一根稻草"嫌疑最大**：exitCode 63 属沙箱/进程启动失败族，而该开关直接动 GPU 沙箱进程；它正是 1.1.2→1.1.5 之间 1607 启动开关的**新增项之一**。
- 也可能是 compat 开关组合、或 safe-mode 机制本身的副作用。**无真机逐开关测试，无法 100% 锁定单一开关。**

## 7. 关于"1.1.3 崩"的辨析

- **代码事实**：1.1.3（`1d5fc8a`）的 1607 启动配置与 1.1.2 **完全相同**。`cc20b06`（1.1.3 区间）虽然加了 `--disable-gpu`，但被 `if(shouldDisableHardwareAcceleration)` 守卫，**1607（=false）吃不到**。
- **结论**：若 1.1.3 确实崩，**原因不在 main.js 启动参数**——可能是 `cc20b06` 同时改的 `content-preload.js`(34 行) 或区间其他改动引入的**另一种**白屏/页面问题，与当前 `launch-failed` 非同一机制；也可能用户对版本号记忆有偏差，真正的 launch-failed 回归在 **1.1.4（`cd40a25` 禁 GPU）**。
- **建议**：复核 1.1.3 的具体表现（是"白屏但进程在"，还是"launch-failed 崩溃"），以区分这两层问题。

## 8. 改代码避坑指南（Do / Don't）

### ✅ 推荐方向
- **让 Win10 1607 回到"正常 Win10 路径"（= 1.1.2 行为）**：在 `getLegacyWindowsGpuWorkaroundInfo()` 里**移除 `build<=14393` 这条特殊分支**，让 1607 落入正常 `return {false,false}`。这是被生产环境证实能用的配置。
- 若担心 1607 真有白屏，**先在真机逐项验证**再加开关，且优先怀疑 `disable-gpu-sandbox`。

### ⛔ 不要做（每条都有血的教训）
1. **不要对 Win10 1607 禁 GPU**（`--disable-gpu` / `in-process-gpu`）——现场已多次证实 1607 禁 GPU → `launch-failed`（cd40a25 教训 + main.js:37 注释）。
2. **不要让 safe-mode 把 1607 切到禁 GPU 重启**——这是把"崩一次"放大成"死循环打不开"的元凶（main.js:661 `maybeRelaunchInRendererSafeMode`，注意它对 1607 `shouldUseRendererLaunchCompatibility=true` 会触发）。
3. **不要轻信"恢复 1.1.2 验证过的参数"这类注释**——经核对，1.1.2 根本没有那套"仅旧系统"的 compat 开关。改动前请 `git show <旧版本>:main.js` 核对真实历史，别凭注释想当然。
4. **不要把"加启动开关"当万能药**——`no-sandbox`/`disable-features` 这类开关在不同 Win10 子版本上行为不一致，叠加越多风险面越大；compat 分支注释也写"避免扩大行为变化面"，但实际却在扩大。

### 🧭 改动纪律
- 任何动到 `getLegacyWindowsGpuWorkaroundInfo()` 或启动开关的改动，**必须区分 Win7/8、Win10 1607、正常 Win10/11 三档分别评估**，且**在真实 1607 机器上回归测试**。
- 这台机器是宝贵的 1607 测试环境，改完务必在它上面验证再发版。

## 9. 只读验证清单（发给现场 / 在真机执行，不改代码即可确诊）

按优先级：
1. **【最高】换运行位置**：便携版整目录复制到 `C:\` 固定目录（非 Temp / 非网络盘）再运行，看白屏是否消失——一步区分"位置/环境"与"GPU/参数"两大方向。
2. **降权运行**：用普通（非 Administrator）账户启动同一便携版，对比是否恢复。
3. **读完整日志**：打开该机"运营助手-本次报错.log"，看 `启动开关:` 那行实际命中哪些开关；数清出现**几条** `render-process-gone`（首次 compat + 二次 safe-mode），对比两次 `reason/exitCode`——确认 safe-mode 二次重启是否也 launch-failed。
4. **临时排除杀软/Defender**：把 exe 及目录加入排除项（或测试机临时停实时防护）再试。
5. **Event Viewer 取证**：Windows 日志→应用程序 看 Application Error 的 faulting module（安装版"已停止工作"尤其要看）；应用程序和服务日志→Microsoft→Windows→AppLocker→EXE and DLL 看有无 8004(blocked)。
6. **对比安装版**：装到 Program Files 运行看是否仍崩，分离"运行位置"与"OS/GPU"。
7. **GPU/驱动核查**：Device Manager 看显卡型号 + 驱动日期；`dxdiag` 确认 WARP/软件渲染是否可用。
8. **网络侧**：用系统浏览器直接访问远程 `scripts-config.json` 的 URL，确认 `ERR_FAILED` 是页面没起来的下游结果，而非企业网拦截。

> `net::ERR_FAILED`（远程配置加载失败）是 **下游症状**：渲染进程没起来，拉远程配置的 fetch 自然失败；代码会回退本地（`script-manager.js`），不致命。

## 10. 已排除的假设（附理由）

| 假设 | 是否成立 | 理由 |
|------|---------|------|
| Electron 21 / Chromium 106 不支持 Win10 1607 | ❌ 排除 | 官方支持矩阵为"Windows 10 and up"，收紧到 Win10-only 是 Electron 23；1607 满足要求 |
| 缺 VC++ 运行库 | ❌ 排除 | 同机 1.1.2 能跑（运行库环境完全相同）；微信等 Chromium 应用也能跑 |
| Electron 版本升级导致 | ❌ 排除 | 1.1.2 与 1.1.5 都是 Electron 21.4.4（区间内仅把 `^21.4.4` 锁成 `21.4.4`，实际版本未变） |
| 系统完全不支持 Chromium / 杀软无差别拦截 | ❌ 排除 | 微信、企业微信（Chromium 内核）同机正常 |
| 便携版"放桌面就好" | ❌ 无效 | portable 是自解压壳，无论放哪都解压到 `%TEMP%` 运行，挪位置不改变实际执行路径 |
| `%TEMP%`+管理员+杀软拦截（环境因素） | 🟡 降级 | 技术上成立，但**被"1.1.2 同环境能用"证伪为主因**；环境没变，变的是代码 |
