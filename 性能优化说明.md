# Session 恢复性能优化说明

## 优化概览

在基础修复（队列串行化）的基础上，添加 **Session 状态缓存**，避免重复恢复相同的 sessionData。

## 工作原理

### 1. Session 指纹计算

为每个 sessionData 计算唯一指纹（基于关键 cookie）：

```javascript
function getSessionDataFingerprint(sessionData) {
  // 提取关键 cookie: sessionid, web_session, BDUSS, pass_ticket 等
  // 计算指纹: name:value前8位的组合
  // 例如: "sessionid:abc12345|web_session:def67890"
}
```

### 2. 缓存状态记录

Session 恢复成功后记录状态：

```javascript
sessionRestoreCache.set(partitionName, {
  fingerprint: "sessionid:abc12345|web_session:def67890",
  timestamp: Date.now()
});
```

### 3. 缓存检查

打开新窗口时先检查缓存：

```javascript
if (isSessionAlreadyRestored(partitionName, effectiveSessionData)) {
  // 指纹相同 且 5分钟内 → 跳过恢复
  console.log('✅ 命中缓存，跳过重复恢复');
  return;
}
```

## 缓存策略

### 缓存命中条件

同时满足以下条件才命中：
1. **相同 partition**（相同账号）
2. **相同指纹**（关键 cookie 值相同）
3. **5分钟内**（缓存未过期）

### 缓存失效条件

以下情况缓存失效，需要重新恢复：
- ❌ 5分钟后（自动过期）
- ❌ 关键 cookie 值变化（如重新授权）
- ❌ 浏览器重启（内存缓存清空）

### 为什么是 5 分钟？

| 时间 | 优点 | 缺点 |
|------|------|------|
| 1分钟 | 数据最新 | 命中率低，频繁发布场景效果差 |
| **5分钟** ✅ | **平衡命中率和新鲜度** | 极端情况可能用旧数据 |
| 30分钟 | 命中率高 | 重新授权后可能用旧登录态 |

## 边界情况处理

### 1. 重新授权场景

**场景**：用户删除授权后重新扫码登录

**处理**：
- 新登录的 sessionid 值改变
- 指纹不匹配，缓存未命中
- 自动走正常恢复流程 ✅

### 2. Cookie 部分变化

**场景**：平台刷新部分 cookie（如 token 续期），但主要登录 cookie（sessionid）不变

**处理**：
- 主要登录 cookie 不变，指纹相同
- 缓存命中，跳过恢复
- **结果**：可能丢失部分更新的 cookie（token）
- **影响**：5 分钟内问题不大，token 有效期通常 >1 小时

### 3. 多个账号快速切换

**场景**：
1. 发布账号A（恢复，记录缓存A）
2. 发布账号B（恢复，记录缓存B）
3. 发布账号A（缓存命中A，跳过）
4. 发布账号B（缓存命中B，跳过）

**处理**：
- 不同 partition 独立缓存
- 互不影响 ✅

### 4. 浏览器重启

**场景**：浏览器关闭后重新打开

**处理**：
- 内存缓存清空
- 首次发布需要恢复（miss）
- 后续发布命中缓存（hit） ✅

## 后续优化方向

### 1. 持久化缓存（可选）

将缓存写入磁盘，重启后仍有效：

```javascript
// 写入 global-storage.json
globalStorage.sessionRestoreCache = {
  'persist:douyin_xxx_1': {
    fingerprint: '...',
    timestamp: 1234567890
  }
};
```

**收益**：浏览器重启后首次发布也能命中缓存
**风险**：可能使用过期登录态

### 2. 智能过期时间

根据平台特性调整缓存时间：

| 平台 | 缓存时间 | 原因 |
|------|---------|------|
| 抖音 | 10分钟 | sessionid 有效期长 |
| 小红书 | 5分钟 | token 刷新频繁 |
| 视频号 | 3分钟 | cookie 去重频繁 |

### 3. 主动预热

在用户选择账号时提前恢复 session：

```javascript
// 首页选择账号时触发
onAccountSelected(accountId) {
  // 后台预热 session，用户点发布时直接可用
  preloadAccountSession(accountId);
}
```

**收益**：发布时 0 等待
**成本**：额外的后台恢复操作
