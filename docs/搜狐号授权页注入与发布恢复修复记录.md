# 搜狐号授权页注入与发布恢复修复记录

> 记录日期：2026-07-14  
> 适用版本：`v1.1.8`  
> 代码提交：`d2d4a17`  
> 主题：搜狐号授权页注入、发布页恢复、发布成功处理与窗口显示异常修复

## 一、修复结论

本次修复解决了搜狐号授权页和发布窗口之间的 URL、窗口上下文与脚本注入冲突，核心结论如下：

1. 浏览器地址栏可以继续显示裸的 `first/page`：

   ```text
   https://mp.sohu.com/mpfe/v4/contentManagement/first/page
   ```

2. 主进程在脚本匹配时，必须把授权窗口的裸 URL 虚拟成：

   ```text
   https://mp.sohu.com/mpfe/v4/contentManagement/first/page?from=auth
   ```

3. `?from=auth` 只用于注入匹配，不强制写回地址栏，避免搜狐 SPA 路由、授权状态和页面自身导航产生额外副作用。

4. 发布窗口只有在当前窗口存在对应的 `publish_data_window_${windowId}` 时，才允许从 `first/page` 恢复到发布页，避免内容管理页和授权页被误判为发布窗口。

5. 发布成功标志必须按窗口隔离读取、立即清理、完成统计上报后再关闭窗口，避免重复上报。

6. 修复 `showManagedWindow()` 使用未定义 `currentURL` 导致的运行时错误，避免窗口在 `ready-to-show` 或超时显示阶段直接抛出 `ReferenceError`。

## 二、用户可见问题

### 1. 授权页脚本没有注入

搜狐授权页实际可能停留在裸 URL：

```text
https://mp.sohu.com/mpfe/v4/contentManagement/first/page
```

虽然 `injected-scripts/scripts-config.json` 已经配置了：

```json
"https://mp.sohu.com/mpfe/v4/contentManagement/first/page?from=auth": [
  "common.js",
  "souhuhao-redirect.js",
  "souhuhao-creator.js"
],
"https://mp.sohu.com/mpfe/v4/contentManagement/first/page*": [
  "common.js",
  "souhuhao-redirect.js",
  "souhuhao-creator.js"
]
```

但配置命中前还存在 `script-manager.js` 的打开页跳过逻辑。如果主进程把裸 URL 原样传入脚本管理器，裸 `first/page` 会被提前跳过，后续配置匹配不会执行。

### 2. 发布窗口误停在 `first/page`

发布页在登录失效、SPA 导航、页面初始化失败或搜狐自身跳转时，可能落到 `first/page`。旧逻辑只看 URL，容易把以下页面混为一谈：

- 内容管理窗口
- 授权窗口
- 发布窗口

如果授权窗口或内容管理窗口被当成发布窗口处理，就会出现错误回跳、页面循环或登录流程被打断。

### 3. 发布成功后重复处理

发布页点击发布后会写入发布成功标志。旧流程在不同存储位置之间读取不一致时，可能出现：

- 成功标志重复读取
- 成功统计重复上报
- 成功窗口未关闭
- 窗口关闭后数据仍残留，影响下一次任务

### 4. 窗口显示阶段运行时错误

修复过程中曾出现以下错误：

```text
ReferenceError: currentURL is not defined
    at showManagedWindow (.../main.js:13128:74)
```

原因是 `showManagedWindow()` 回调没有 `currentURL` 参数，却直接引用了该变量。该问题会影响窗口显示流程，与搜狐脚本本身的业务逻辑无关，但会让整个窗口流程提前抛错。

## 三、根因分析

### 1. 脚本匹配顺序导致配置看似正确但实际不生效

旧调用链为：

```text
窗口导航事件
  -> injectScriptForUrl(currentURL)
  -> scriptManager.getScript(currentURL)
  -> shouldSkipContentManageOpenOnlyUrl(currentURL)
  -> 裸 first/page 直接返回空脚本
```

`script-manager.js` 中存在精确跳过项：

```text
https://mp.sohu.com/mpfe/v4/contentManagement/first/page
```

该判断发生在 manifest/config 精确匹配和通配符匹配之前。因此，单独增加 `first/page*` 配置无法解决裸 URL 被提前跳过的问题。

### 2. 授权窗口上下文可能在导航或复用时丢失

搜狐号窗口会经历以下状态变化：

```text
首页发起授权
  -> 临时 session 授权窗口
  -> mpfe 根入口或 first/page
  -> 登录页 / SPA 路由
  -> first/page 或内容管理页
```

仅通过当前 URL 无法可靠区分授权窗口和发布窗口，需要结合：

- `windowContextMap`
- `purpose: 'auth'` / `purpose: 'publish'`
- `platform: 'sohuhao'`
- `auth_mode_window_${windowId}` 标记
- 当前窗口是否存在 `publish_data_window_${windowId}`

### 3. `toPath` 是跨入口共享状态，容易污染后续导航

搜狐页面会读取 `localStorage.toPath` 进行内部跳转。内容管理入口、授权入口和发布恢复都可能使用同一个 key。如果没有按入口及时清理，会出现：

- 内容管理窗口被带到发布页
- 授权窗口被带到错误页面
- 发布窗口启动时误读上一轮任务的目标路径

因此本次增加入口标记并在入口阶段清理 `toPath`。

## 四、实现方案

### 1. `main.js`：统一搜狐授权入口和注入匹配

新增或调整以下逻辑：

#### `normalizeSohuhaoAuthEntryUrl()`

作用：

- 仅处理 `platform=sohuhao` 且 `purpose=auth` 的窗口。
- 将搜狐 mpfe 根入口和裸 `first/page` 规范为 `first/page?from=auth`。
- 保留原有查询参数，但覆盖 `from` 为 `auth`。
- 只在主进程内部用于加载或匹配，不强制改变用户看到的业务 URL。

#### `isSohuhaoAuthInjectablePageUrl()`

识别搜狐授权相关入口：

- `https://mp.sohu.com/mpfe/v4/`
- `https://mp.sohu.com/mpfe/v4/login...`
- `https://mp.sohu.com/mpfe/v4/contentManagement/first/page...`

#### `inferSohuhaoAuthWindowContext()`

当窗口上下文不完整时，根据以下条件推断授权窗口：

- 已有 `purpose: 'auth'`
- 当前 URL 属于搜狐授权可注入页面
- `globalStorage` 中存在 `auth_mode_window_${windowId}` 标记

推断结果补充：

```js
{
  purpose: 'auth',
  platform: 'sohuhao',
  expectedPageUrl: 'https://mp.sohu.com/mpfe/v4/contentManagement/first/page'
}
```

#### `getSohuhaoAuthInjectionMatchUrl()`

这是主进程注入链路的关键函数。它将：

```text
实际页面 URL：first/page
```

转换成：

```text
脚本匹配 URL：first/page?from=auth
```

随后再调用统一的 `injectScriptForUrl()`。这样可以绕过裸 `first/page` 的打开页跳过规则，同时不需要强制修改地址栏。

#### 注入事件统一使用虚拟匹配 URL

以下事件均使用 `getSohuhaoAuthInjectionMatchUrl()`：

- `dom-ready`
- `did-finish-load`
- `did-navigate-in-page`

这样可以避免首次加载、完整导航和 SPA 路由导航分别采用不同匹配规则。

#### 授权窗口复用前的 URL 规范化

`routeSohuhaoAuthDataToAccountWindow()` 在复用授权窗口前调用：

```text
ensureSohuhaoAuthWindowInjectableUrl()
```

如果复用窗口当前仍停留在不可注入的根入口或裸 `first/page`，主进程会先导航到可注入入口，再投递授权消息。

#### `get-window-context` 返回推断后的上下文

页面侧通过 `window.browserAPI.getWindowContext()` 获取上下文时，主进程会结合当前 URL 和授权窗口标记返回推断结果，保证 preload 和平台脚本能区分授权窗口。

#### `showManagedWindow()` 变量修复

窗口显示回调内不再使用不存在的外部变量，而是安全读取当前页面 URL：

```js
const currentURL = !newWindow.isDestroyed() && !newWindow.webContents.isDestroyed()
  ? newWindow.webContents.getURL()
  : '';
```

这样即使窗口在销毁边界触发显示回调，也不会因为 `currentURL` 未定义直接抛出异常。

### 2. `content-preload.js`：增加授权页注入兜底

preload 兜底只在以下条件同时满足时执行：

1. 当前 URL 是搜狐根入口或 `first/page`。
2. `get-window-context` 返回 `platform=sohuhao`。
3. `purpose=auth`。
4. 页面中尚未发现 `souhuhao-creator.js` 已注入标记。

兜底通过 IPC 请求虚拟匹配 URL：

```text
https://mp.sohu.com/mpfe/v4/contentManagement/first/page?from=auth
```

然后在页面上下文执行返回的脚本。该逻辑是主进程注入失败时的第二道防线，不应替代主进程的统一匹配逻辑。

### 3. `injected-scripts/scripts-config.json`：补齐搜狐入口规则

搜狐相关规则包括：

```text
https://mp.sohu.com/mpfe/v4
https://mp.sohu.com/mpfe/v4/
https://mp.sohu.com/mpfe/v4/contentManagement/first/page?from=content
https://mp.sohu.com/mpfe/v4/contentManagement/first/page?from=auth
https://mp.sohu.com/mpfe/v4/contentManagement/first/page*
https://mp.sohu.com/mpfe/v4/login*
https://mp.sohu.com/mpfe/v4/contentManagement/news/addarticle*
```

脚本顺序遵循：

```text
common.js
souhuhao-redirect.js
souhuhao-creator.js 或 souhuhao-publish.js
```

### 4. `souhuhao-redirect.js`：分离内容入口、授权入口和发布恢复

#### 内容入口 `from=content`

进入内容管理页时：

- 清理 `localStorage.toPath`。
- 写入 `__sohuhao_content_entry__`。
- 删除地址栏中的 `from=content`，避免页面后续重复识别入口。

#### 授权入口 `from=auth`

进入授权页时：

- 清理 `localStorage.toPath`。
- 写入 `__sohuhao_auth_entry__`。
- 保留 `from=auth`，便于后续 SPA 导航和注入链路识别。

#### 根入口 `/mpfe/v4/`

如果根入口窗口不是发布窗口，则跳转到：

```text
https://mp.sohu.com/mpfe/v4/contentManagement/first/page?from=auth
```

如果根入口属于发布窗口，则跳过授权入口兜底，避免发布窗口被误导航。

#### 发布页恢复

只有当前窗口读取到：

```text
publish_data_window_${windowId}
```

并且当前 URL 确实是 `first/page` 时，才会恢复到发布页。恢复次数通过 sessionStorage 限制为最多 3 次，避免循环跳转。

#### 发布成功处理

发布成功标志支持以下来源：

- `globalStorage.PUBLISH_SUCCESS_DATA_${windowId}`
- `localStorage.PUBLISH_SUCCESS_DATA_${windowId}`
- `localStorage.PUBLISH_SUCCESS_DATA`
- `localStorage.sohu_publish_success_data`

处理顺序：

1. 读取并解析成功标志。
2. 获取 `publishId`。
3. 立即清理 localStorage 和 globalStorage 标志。
4. 调用 `sendStatistics()` 或手动上报兜底。
5. 调用 `closeWindowWithMessage()` 或 `sendMessageToParent()`。

### 5. `souhuhao-publish.js`：限制发布页恢复范围

发布脚本侧增加以下保护：

- 有 `publish-data` 或窗口上下文为 `publish` 时，不执行内容入口回跳。
- 只有带有新鲜内容入口标记，或 referrer 明确来自 `from=content` 时，才返回内容管理页。
- 监听 SPA 的 `pushState`、`replaceState`、`popstate`、`hashchange` 和定时检查。
- 连续缺少必需元素时，只有当前窗口仍有发布数据才触发恢复。
- 对发布页乱码渲染和白屏分别设置检测与刷新上限。

## 五、完整运行流程

### 授权流程

```text
首页发起搜狐授权
  -> 打开临时授权窗口
  -> 标记 auth_mode_window_${windowId}
  -> 页面落到根入口或裸 first/page
  -> 推断窗口上下文为 sohuhao/auth
  -> 注入匹配 URL 转换为 first/page?from=auth
  -> common.js + souhuhao-redirect.js + souhuhao-creator.js
  -> 用户完成授权
  -> 保存授权数据并关闭或复用窗口
```

### 发布流程

```text
首页发起发布
  -> 打开发布窗口
  -> 写入 publish_data_window_${windowId}
  -> 进入 addarticle
  -> 注入 common.js + souhuhao-publish.js
  -> 页面异常落到 first/page
  -> 读取同窗口 publish-data
  -> 写入 toPath 并恢复 addarticle
  -> 发布成功写入 PUBLISH_SUCCESS_DATA_${windowId}
  -> 成功页读取、清理、上报、通知首页并关闭窗口
```

### 内容管理流程

```text
首页打开内容管理
  -> 使用 from=content 入口
  -> 清理旧 toPath
  -> 写入内容入口标记
  -> 进入内容管理页
  -> 不因缺少 publish-data 而回跳发布页
```

## 八、风险与边界

1. `?from=auth` 是注入匹配标记，不代表必须让地址栏一直保留该参数。
2. `content-preload.js` 兜底依赖 `get-window-context()` 返回正确的授权上下文。
3. 发布页恢复必须依赖当前窗口自己的 `publish_data_window_${windowId}`，不能只依据 URL。
4. 发布恢复次数有上限，超过上限后停止自动回跳，避免死循环。
5. 发布成功标志清理发生在上报前，清理失败不会阻止成功上报和窗口关闭。
6. 本文记录的是脚本注入和窗口流程修复，不代表搜狐平台接口、验证码或账号登录状态永久稳定。
7. 修改脚本配置后需要重启 Electron 应用，已有进程不会自动加载新的 `main.js` 或配置文件。
