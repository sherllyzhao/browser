# 操作痕迹审计与优化计划

更新时间：2026-06-10

## 一、审计口径

这是什么：本文只审计注入脚本里的浏览器操作痕迹，不讨论内容是否像 AI 生成。

为什么要这么做：平台页面能观察到的不只是等待时间，还包括事件来源、DOM 改写方式、文件上传方式、焦点变化、重试节奏和窗口生命周期。

为什么这是好主意：先把风险按代码位置排清楚，后续优化才能集中在高影响文件，避免在低风险脚本里做无效改动。

扫描范围：`injected-scripts/*.js`

重点模式：

- `dispatchEvent(...)`
- `.click(...)`
- `setNativeValue(...)`
- `new MouseEvent/InputEvent/KeyboardEvent/ClipboardEvent/...`
- `DataTransfer` / `.files =`
- `innerHTML =` / `.value =`
- `.focus()` / `.blur()`
- 固定或集中式 `setTimeout(...)` / `delay(...)`

这些脚本主要做授权、登录检测、页面状态检查、关闭窗口或提示展示，操作痕迹少于发布脚本。

| 文件类型 | 示例 | 当前判断 |
| --- | --- | --- |

## 三、已完成的第一轮优化

这是什么：已将大多数固定等待改为公共随机等待，新增 `getRandomDelayMs(...)` / `delay(...)` 作为统一入口。

为什么要这么做：固定等待会让所有平台脚本表现出相同节奏，后续维护也容易出现一处改、一处漏。

为什么这是好主意：随机等待统一在公共层，既能减少重复代码，也能让后续调参集中在 `common.js`，降低维护成本。

已完成：

- `common.js` 新增 `getRandomDelayMs(...)`
- `delay(...)` 改为基于 `getRandomDelayMs(...)`
- 多数 `await new Promise(setTimeout...)` 已收口为 `window.delay(...)`
- 多数数字 `setTimeout(...)` 已接入随机等待

## 四、下一步优化顺序

### 1. 公共等待表达式收口

这是什么：把 common-backed 脚本里重复的长 fallback 表达式收口为 `window.getRandomDelayMs(...)`。

为什么要这么做：当前批量生成的表达式过长，后续读代码和排查问题不方便。

为什么这是好主意：不改变运行行为，只降低维护成本和误改概率。

### 2. 操作封装分层

这是什么：逐步把标题输入、正文填充、封面上传、定时发布等高频动作抽象成可复用 helper。

为什么要这么做：现在每个平台各自维护一套 `focus -> setNativeValue -> dispatchEvent -> delay`，重复多且差异难审。

为什么这是好主意：可以减少无意义的重复事件触发，也能让失败处理和日志更一致。

### 3. 减少不必要的重复事件

这是什么：在高风险发布脚本中，逐步审查连续触发的 `input/change/keyup/blur` 是否都必要。

为什么要这么做：过密的合成事件会增加操作链复杂度，也让页面状态更难判断。

为什么这是好主意：减少冗余事件能提升稳定性，降低因为页面框架升级导致的误触发。

### 4. 保留合规边界

这是什么：本文不提供绕过平台检测的伪装方案，不做事件可信度伪造或规避风控的实现。

为什么要这么做：第三方平台可能有自己的自动化和内容发布规则，绕过检测会带来账号和业务风险。

为什么这是好主意：优化目标应是稳定、可维护、用户授权明确，而不是扩大不可控风险。
