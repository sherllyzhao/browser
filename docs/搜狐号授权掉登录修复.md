# 搜狐号授权后掉登录 / 显示错账号 修复说明

## 一句话结论

搜狐号是**多账号模式**，但**授权脚本把 cookies 写错了 session**：写到了全平台共享的主 session（`persist:browserview`），而发布窗口读的是账号独立 session（`persist:souhuhao_<account_info.id>`）。两边对不上，导致发布时要么掉登录、要么用到残留的其他账号。

修复方式：授权成功后把 cookies 迁移到**发布窗口真正会读取的那个账号 session**。

---

## 现象

1. 在搜狐授权页扫码授权完成 → 刚打开发布页就又提示未登录（掉登录）。
2. 之前尝试修复时改成「信任本地 session」→ 客户的号显示成了操作者自己的账号（串号）。
3. 再改回来 → 又变回保存不了登录。

三个现象其实是**同一个根因的两面**，在错误的 session 上无论怎么调「本地优先 / sessionData 优先」策略都无法稳定命中。

---

## 数据流梳理

### 授权阶段（`injected-scripts/souhuhao-creator.js`）

- 授权窗口使用**临时 session**（`openNewWindow(..., { useTemporarySession: true })`）。
- 授权成功后做两件事：
  1. `getFullSessionData('mp.sohu.com')` → 完整会话上传后台 `/api/mediaauth/shinfo`。
  2. 迁移 cookies，让本地发布时可用。

### 发布阶段（`content-preload.js` → `main.js`）

- 发布窗口使用**账号独立 session**：`getAccountSession('souhuhao', accountId)` → 分区名 `persist:souhuhao_<accountId>`（`main.js` 中 `getAccountSession`）。
- 其中 `accountId` = 后台返回的 `element.account_info.id`（`content-preload.js` 中 `buildPublishWindowJob` 的 `backendAccountId`）。

### 冲突点

| 环节 | 读写的 session |
|------|----------------|
| 授权脚本（修复前） | `migrateCookiesToPersistent('mp.sohu.com')` → 写 **`persist:browserview`**（主 session） |
| 发布窗口 | 读 **`persist:souhuhao_<account_info.id>`**（账号独立 session） |

授权写的 session 和发布读的 session **根本不是同一个**，所以：

- **走 sessionData 覆盖**：账号 session 里没有授权写入的 cookie，只能靠后台回传那份 `element.cookies` 恢复；只要那份不完整 / 过期，就**掉登录**。
- **信任本地 session**（`main.js` 的「本地优先策略」跳过覆盖）：账号 session 可能残留上一次别的账号的登录态 → **串号**。

---

## 修复内容

只改了 `injected-scripts/souhuhao-creator.js`，做法对齐了已经验证正常的腾讯号 `injected-scripts/tengxvnhao-creator.js`：

1. 新增常量与辅助函数：
   - `SOUHUHAO_SESSION_DOMAINS = ['sohu.com', 'mp.sohu.com']`：迁移时按这些域名过滤 cookie。
   - `getPublishWindowData()`：读取 `publish_data_window_<windowId>`，用来判断当前授权是否发生在某个发布窗口内（即「掉登录后在发布窗口里重新扫码」）。
2. 授权成功后缓存 `cachedPublishWindowData`。
3. 迁移逻辑改为：
   - **有发布账号上下文**（`publishData.element.account_info.id` 存在）→ 调 `migrateCookiesToAccountSession(domain, 'souhuhao', accountId)`，直接写回发布窗口读取的 `persist:souhuhao_<accountId>`。
   - **无发布账号上下文**（首次独立授权）→ 回退到原来的 `migrateCookiesToPersistent(domain)`。

### 涉及的 API

- `window.browserAPI.migrateCookiesToAccountSession(domain, platform, accountId)`
  - 对应主进程 handler：`migrate-cookies-to-account-session`（`main.js`）。
  - 该 handler 以传入的 `domain` 为准过滤 cookie，不依赖任何硬编码平台 map，因此搜狐可直接使用。

---

## 为什么不改 main.js 的「本地优先策略」

`main.js` 里打开发布窗口时的「本地优先策略」（本地已有有效登录态则跳过 sessionData 覆盖）逻辑本身没问题，问题出在**授权阶段把 cookies 写错了 session**。源头修好后，本地策略就能正常工作，无需改动，避免再次反复横跳。

---

## 覆盖场景与残留风险

| 场景 | 是否被本次修复覆盖 |
|------|------------------|
| 掉登录后在**发布窗口内重新扫码授权** | ✅ 直接写回账号 session，发布不再掉登录 / 串号 |
| 首次在**授权页（AddAccount.vue）独立授权** | ⚠️ 走回退分支（写 `persist:browserview`）。此路发布时仍依赖后台回传的 `element.cookies`（授权时已上传 `/api/mediaauth/shinfo`，发布时 `openNewWindow` 用 `element.cookies` 恢复）。能否稳住取决于后台那份完整会话数据是否完整。 |

---

## 后续可选优化（本次未做）

`main.js` 中 `migrate-to-new-account` handler 内**硬编码**的 `platformDomains` map 缺少 `sohuhao` 与 `baijiahao`，与 `domain-config.js` 中的 `platformDomains` 不同步。本次修复未走该路径，暂不受影响；但建议后续将其统一到 `domain-config.js`，避免以后新增平台走该 handler 时踩坑。
