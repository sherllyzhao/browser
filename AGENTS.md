# Repository Guidelines

## Project Structure & Module Organization
- `main.js`: Electron main process (window lifecycle, session, IPC, storage).
- `preload.js` and `content-preload.js`: renderer bridge APIs exposed to pages.
- `index.html`, `login.html`, `not-*.html`: local UI pages loaded by Electron.
- `renderer.js`: control-panel rendering and page interaction logic.
- `script-manager.js`: injection orchestration and script loading.
- `injected-scripts/`: platform automation scripts and config.
  Example naming: `douyin-creator.js`, `zhihu-publish.js`, `scripts-config.json`.
- `assets/`: icons and static UI assets. `docs/`: implementation notes.
- `dist/`: build artifacts generated by `electron-builder`.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm start`: run locally with Electron (`electron .`).
- `npm run build`: create Windows installer output in `dist/`.
- `npm run build:dir`: unpacked Windows build for inspection.
- `npm run build:portable`: build portable Windows package (ia32 target).

## Coding Style & Naming Conventions
- JavaScript (CommonJS) with semicolons and 2-space indentation.
- Prefer `const`/`let`; keep functions focused and side effects explicit.
- Use `camelCase` for variables/functions, `UPPER_SNAKE_CASE` for constants.
- Keep platform scripts in `injected-scripts/` with `<platform>-<action>.js`.
- Keep logs actionable and scoped (for example: `[Global Storage] ...`).

## Testing Guidelines
- No automated test runner is currently configured (`npm test` is not defined).
- Validate changes with targeted manual smoke tests via `npm start`:
  - login flow and window open/close behavior
  - script injection on mapped URLs
  - IPC messaging between home and child windows
  - session persistence after app restart
- For script changes, test at least one success and one failure path per platform.
- After every JavaScript edit, explicitly verify that any newly introduced runtime variables/functions are defined before use. For injected scripts, add a symbol-existence self-check (`typeof ...`) or equivalent guard on critical paths before considering the change complete.

## Commit & Pull Request Guidelines
- Follow Conventional Commit prefixes seen in history: `feat:`, `fix:`, `chore:`.
- Keep each commit focused on one change set.
- PRs should include:
  - clear summary and impacted modules
  - related issue/task link (if available)
  - manual verification steps and results
  - screenshots/GIFs for UI or flow changes (`login.html`, publish pages, etc.)

## Security & Configuration Tips
- Do not commit real cookies/tokens or production account data.
- Review `config.js` and `injected-scripts/scripts-config.json` before release.
- Treat session/cookie handling changes as high risk; document rollback steps in PR.
